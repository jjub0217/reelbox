# 스프린트 4 테스트 시나리오

스프린트 4의 목표는 태그가 누적되어도 사용자가 직접 정리할 수 있는지 확인하는 것이다.

## 테스트 범위

- 사이드바에서 `태그 관리` 페이지 진입
- 새 태그 직접 추가
- 태그명 수정
- 태그 삭제
- 태그 입력 시 정규화 규칙 적용

## 사전 조건

- 개발 서버가 실행 중이어야 한다
- 테스트 계정으로 로그인 가능해야 한다
- 릴스를 1개 이상 저장할 수 있어야 한다

## 테스트 시나리오 1

이름
- `태그 관리` 페이지 진입 확인

목적
- 사이드바에서 태그 관리 화면으로 자연스럽게 이동할 수 있는지 확인한다

사전 조건
- 로그인된 상태여야 한다

실행 순서
1. 우측 상단 메뉴 버튼을 눌러 사이드바를 연다
2. `태그 관리` 메뉴를 누른다
3. 이동한 화면의 제목과 목록 영역을 확인한다

기대 결과
- `태그 관리` 페이지로 이동한다
- 태그가 없으면 빈 상태 문구가 보인다

체감 포인트
- 카테고리 관리와 비슷한 구조로 이해되는가
- 메뉴 이름이 직관적인가

## 테스트 시나리오 2

이름
- 새 태그 추가 확인

목적
- 태그를 관리 화면에서 직접 추가할 수 있는지 확인한다

사전 조건
- `태그 관리` 페이지에 들어와 있어야 한다

실행 순서
1. 새 태그 입력창에 `#맛집` 또는 ` 디저트 ` 같은 값을 입력한다
2. `추가` 버튼을 누른다
3. 목록에 반영된 태그명을 확인한다

기대 결과
- 태그가 추가된다
- 저장된 태그명은 정규화되어 보인다

체감 포인트
- 태그를 따로 관리 화면에서 추가하는 흐름이 필요한가
- 입력 규칙이 예측 가능한가

## 테스트 시나리오 3

이름
- 태그명 수정 확인

목적
- 오탈자나 표현을 태그 관리 화면에서 수정할 수 있는지 확인한다

사전 조건
- 태그가 1개 이상 있어야 한다

실행 순서
1. 목록에서 아무 태그나 `수정`을 누른다
2. 태그명을 다른 값으로 바꾼다
3. `저장`을 누른다
4. 목록 반영 상태를 확인한다

기대 결과
- 수정된 태그명이 목록에 반영된다
- 이미 존재하는 태그명으로는 수정할 수 없다

체감 포인트
- 수정 UI가 단순한가
- 카테고리 관리와 동일한 패턴이라 익숙한가

## 테스트 시나리오 4

이름
- 태그 삭제 확인

목적
- 태그를 안전하게 삭제할 수 있는지 확인한다

사전 조건
- 태그가 1개 이상 있어야 한다

실행 순서
1. 목록에서 아무 태그나 `삭제`를 누른다
2. 확인창 문구를 확인한다
3. 삭제를 진행한다
4. 목록에서 태그가 사라졌는지 확인한다

기대 결과
- 태그가 삭제된다
- 삭제 확인 문구가 이해 가능해야 한다

체감 포인트
- 서비스 화면 기준으로 충분히 단순한가
- 숫자 정보 없이도 삭제 판단이 가능한가

## 테스트 시나리오 5

이름
- 릴스 입력 화면과 태그 관리 규칙 일관성 확인

목적
- 릴스 저장 화면의 태그 입력과 태그 관리 화면의 정규화 규칙이 일관된지 확인한다

사전 조건
- 새 릴스를 저장할 수 있어야 한다

실행 순서
1. `+ 릴스 추가` 화면으로 이동한다
2. 태그에 `#맛집`, `맛집`, `  맛집  ` 중 하나 이상을 입력한다
3. 저장한다
4. 다시 `태그 관리` 페이지로 이동한다
5. 목록에서 태그명이 어떻게 저장됐는지 확인한다

기대 결과
- 같은 의미의 입력은 동일한 정규화 결과로 저장된다
- 중복 태그가 불필요하게 늘어나지 않아야 한다

체감 포인트
- 태그 입력 규칙이 서비스 전체에서 일관적인가
- 사용자가 규칙을 몰라도 자연스럽게 쓸 수 있는가

관찰 메모

- 새 태그를 추가했을 때 바로 눈에 보여야 해서 검색과 추가 영역은 목록 위에 있어야 함
- `이미 추가된 태그입니다` 문구는 새 태그 추가 영역 바로 아래에 보이는 편이 자연스러움
- 중복 태그 안내는 입력값을 수정하기 시작하면 바로 사라지는 편이 더 자연스러움
