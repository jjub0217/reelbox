# 스프린트 5 테스트 시나리오

스프린트 5의 목표는 핵심 기능의 최소 검증 범위를 정리하고, 운영 안정성과 썸네일 fallback이 실제로 안전하게 동작하는지 확인하는 것이다.

## 테스트 범위

- 릴스 생성
- 릴스 수정
- 검색
- 소유권 검증
- 카테고리 CRUD
- 썸네일 추출 실패 시 fallback
- 잘못된 인스타 기본 이미지 차단

## 사전 조건

- 개발 서버가 실행 중이어야 한다
- 테스트 계정으로 로그인 가능해야 한다
- 가능하다면 계정 2개로 접근 제한 테스트를 다시 확인할 수 있어야 한다

## 이번 스프린트의 최소 회귀 테스트 기준

아래 항목은 이후 기능 변경이 생겨도 최소한 다시 확인해야 하는 기준이다.

- 릴스를 정상 저장할 수 있어야 한다
- 저장된 릴스를 수정할 수 있어야 한다
- 메모, 후기, 태그, 카테고리 기준 검색이 동작해야 한다
- 다른 사용자의 저장 레코드 상세에 접근할 수 없어야 한다
- 카테고리를 생성/수정/삭제할 수 있어야 한다
- 썸네일이 실패해도 저장과 조회 흐름이 끊기지 않아야 한다

## 테스트 시나리오 1

이름
- 릴스 생성 기본 동작 확인

목적
- 릴스를 저장하는 가장 기본 흐름이 정상인지 확인한다

사전 조건
- 로그인된 상태여야 한다

실행 순서
1. `+ 릴스 추가` 화면으로 이동한다
2. 저장 가능한 인스타그램 URL을 입력한다
3. 카테고리, 태그, 메모 또는 후기를 일부 입력한다
4. 저장한다
5. 홈에서 방금 저장한 릴스를 확인한다

기대 결과
- 릴스가 저장된다
- 홈 목록에 방금 저장한 항목이 보인다

## 테스트 시나리오 2

이름
- 릴스 수정 기본 동작 확인

목적
- 저장된 릴스의 주요 필드를 수정할 수 있는지 확인한다

사전 조건
- 기존 릴스가 1개 이상 있어야 한다

실행 순서
1. 릴스 상세 페이지로 이동한다
2. 수정 화면으로 이동한다
3. 메모, 후기, 카테고리, 태그 중 하나 이상을 바꾼다
4. 저장한다
5. 상세 페이지에서 변경 내용을 확인한다

기대 결과
- 수정 내용이 반영된다
- 후기를 입력했다면 방문 완료 상태도 함께 반영된다

## 테스트 시나리오 3

이름
- 검색 기본 동작 확인

목적
- 메모, 후기, 태그, 카테고리 기준 검색이 계속 정상인지 확인한다

사전 조건
- 검색 가능한 테스트 데이터가 있어야 한다

실행 순서
1. 메모에만 들어 있는 단어로 검색한다
2. 후기에만 들어 있는 단어로 검색한다
3. 태그명으로 검색한다
4. 카테고리명으로 검색한다

기대 결과
- 각 경우에 해당 릴스가 검색된다

## 테스트 시나리오 4

이름
- 소유권 검증 확인

목적
- 다른 사용자의 저장 레코드에 접근할 수 없는지 확인한다

사전 조건
- 계정 A와 계정 B가 있어야 한다

실행 순서
1. 계정 A로 릴스를 저장한다
2. 계정 A의 상세 페이지 URL을 복사한다
3. 계정 B로 로그인한다
4. 계정 B에서 해당 URL로 직접 접속한다

기대 결과
- 계정 B에서는 404 또는 not found로 보여야 한다

## 테스트 시나리오 5

이름
- 카테고리 CRUD 확인

목적
- 카테고리 생성, 수정, 삭제가 계속 정상인지 확인한다

사전 조건
- 로그인된 상태여야 한다

실행 순서
1. 사이드바에서 `카테고리 관리`로 이동한다
2. 새 카테고리를 추가한다
3. 방금 만든 카테고리명을 수정한다
4. 해당 카테고리를 삭제한다

기대 결과
- 생성, 수정, 삭제가 모두 정상 반영된다

## 테스트 시나리오 6

이름
- 썸네일 추출 실패 시 fallback 확인

목적
- 썸네일 추출이 실패해도 사용자 흐름이 끊기지 않는지 확인한다

사전 조건
- 썸네일 추출 실패 가능성이 있는 인스타 URL을 준비해야 한다

실행 순서
1. 릴스 추가 화면에서 썸네일 추출 실패 사례 URL을 입력한다
2. 미리보기 영역이 어떻게 보이는지 확인한다
3. 저장한다
4. 홈 카드와 상세 페이지를 확인한다

기대 결과
- 작성 화면에서 안내 문구가 포함된 fallback이 보인다
- 저장은 계속 가능하다
- 홈/상세에서도 깨진 이미지 대신 fallback UI가 보인다

## 테스트 시나리오 7

이름
- 잘못된 인스타 기본 이미지 차단 확인

목적
- 인스타 로고 같은 기본 이미지가 썸네일처럼 보이지 않는지 확인한다

사전 조건
- 과거에 인스타 기본 이미지가 보였던 사례가 있거나, 새로 실패 사례를 만들 수 있어야 한다

실행 순서
1. 홈 목록에서 썸네일이 없는 릴스 또는 실패 사례 릴스를 확인한다
2. 인스타 로고, favicon, 기본 아이콘 이미지가 그대로 노출되는지 확인한다
3. 상세 페이지에서도 같은 항목을 확인한다

기대 결과
- 인스타 기본 이미지가 썸네일처럼 보이면 안 된다
- 모두 동일한 fallback 규칙으로 보여야 한다

## 운영 메모 확인 항목

이 스프린트에서는 구현 테스트와 별개로 아래 운영 메모도 정리 대상이다.

- 관리자 통계는 어떤 데이터 소스를 기준으로 계산하는가
- 탈퇴 사용자는 앱 DB와 auth 시스템에서 각각 어떻게 취급되는가
- 썸네일 실패는 `추출 실패`와 `저장된 URL 만료`를 어떻게 구분할 것인가

- 카테고리 관리에서 생성/삭제 이후 에러 문구가 남는 문제가 있었고, 진행 상태와 에러 정리 로직을 보강함
- 인스타 기본 이미지나 추출 실패 케이스는 공통 fallback 규칙으로 정리되어야 함
