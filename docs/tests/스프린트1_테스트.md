# 스프린트 1 테스트 시나리오

스프린트 1의 목표는 아래 3가지를 검증하는 것이다.
- 후기 검색이 실제로 동작하는지
- 인스타그램 URL 검증 및 정규화가 기대대로 동작하는지
- 저장 레코드 상세 접근이 계정 기준으로 분리되는지

## 테스트 범위

- 검색 대상에 `review`가 포함되었는지
- 동일 릴스를 URL 형태만 바꿔도 중복 저장이 막히는지
- 인스타그램 릴스 형식이 아닌 URL은 차단되는지
- 다른 계정이 내 저장 레코드 상세에 접근할 수 없는지

## 사전 조건

- 개발 서버가 실행 중이어야 한다
- 테스트 가능한 사용자 계정이 최소 2개 있어야 한다
- 계정 A와 계정 B 모두 로그인 가능해야 한다
- 계정 A는 테스트용 릴스를 저장할 수 있어야 한다

## 테스트 시나리오 1

이름
- 후기 텍스트 검색

목적
- 메모가 아닌 후기 텍스트로도 검색이 되는지 확인한다

사전 조건
- 계정 A로 로그인되어 있어야 한다

실행 순서
1. 새 릴스를 저장한다
2. `메모`는 비워두거나 검색에 쓰지 않을 문구를 넣는다
3. `후기`에 고유한 문구를 입력한다
4. 예시: `연남동 브런치 재방문 예정`
5. 저장 후 홈으로 이동한다
6. 검색창에 `브런치` 또는 `재방문`을 입력한다

기대 결과
- 방금 저장한 릴스가 검색 결과에 노출된다
- 검색창 placeholder에 `후기`가 포함되어 있다

비고
- 가능하면 흔한 단어보다 고유한 단어를 사용한다

## 테스트 시나리오 2

이름
- 동일 릴스 URL 정규화 및 중복 저장 방지

목적
- 같은 인스타그램 릴스를 URL 형태만 바꿔 저장해도 중복 저장이 막히는지 확인한다

사전 조건
- 계정 A로 로그인되어 있어야 한다
- 저장 가능한 인스타그램 릴스 URL이 하나 준비되어 있어야 한다

실행 순서
1. 아래 형태 중 하나로 릴스를 저장한다
2. 예시 1: `https://www.instagram.com/reel/ABC123/`
3. 같은 릴스에 대해 다른 형태의 URL로 다시 저장을 시도한다
4. 예시 2: `https://instagram.com/reel/ABC123`
5. 또는 쿼리스트링이 포함된 URL로 다시 저장을 시도한다
6. 예시 3: `https://www.instagram.com/reel/ABC123/?igsh=sample`

기대 결과
- 첫 저장은 정상 저장된다
- 두 번째 저장은 `이미 저장된 릴스입니다` 메시지로 차단된다

비고
- 실제 테스트에서는 동일한 릴스 shortcode를 사용해야 한다

## 테스트 시나리오 3

이름
- 지원하지 않는 URL 차단

목적
- 인스타그램 릴스 저장 대상이 아닌 URL이 입력되면 저장이 차단되는지 확인한다

사전 조건
- 계정 A로 로그인되어 있어야 한다

실행 순서
1. 새 릴스 추가 화면으로 이동한다
2. 아래 URL을 각각 입력하고 저장을 시도한다
3. `https://www.google.com`
4. `https://www.instagram.com/explore/`
5. `abcd`

기대 결과
- 저장이 되지 않아야 한다
- `올바른 인스타그램 릴스 URL을 입력해주세요` 메시지가 표시되어야 한다

비고
- 현재 허용 대상은 `reel`, `reels`, `p` 형식이다

## 테스트 시나리오 4

이름
- 계정 간 저장 레코드 상세 접근 제한

목적
- 다른 계정이 내 저장 레코드 상세 페이지를 직접 열 수 없는지 확인한다

사전 조건
- 계정 A와 계정 B가 준비되어 있어야 한다

실행 순서
1. 계정 A로 로그인한다
2. 릴스를 하나 저장한다
3. 저장된 릴스 상세 페이지 URL을 복사한다
4. 예시: `/reels/저장레코드ID`
5. 로그아웃한다
6. 계정 B로 로그인한다
7. 복사한 URL로 직접 접속한다

기대 결과
- 계정 B는 계정 A가 저장한 해당 상세 페이지를 열 수 없어야 한다
- 화면은 404 또는 not found 흐름으로 보여야 한다

비고
- 여기서 막는 대상은 인스타 원본 URL이 아니라 ReelBox 내부 저장 레코드다
